## What this run is

`llama32-1b-epoch3-v3` (cato) is the Tinker run with id
`0951cfea-85a6-5a21-910e-fa0b646b9d5d`. It produced the merged
safetensors directory at
`cato:/home/cato-user/tinker_merged/llama32-1b-epoch3-v3/` and the
single-file step-10500 copy at `..._step10500/`. As of 2026-05-17
18:02 it is served by `vllm serve` on cato port `:19320` under the
served-model-name `llama32_1b_epoch3`. The flags are
`--max-num-seqs 48 --max-num-batched-tokens 24576 --enforce-eager`.

It is the production planner baseline. The number it is asked to
beat is its own val_loss of 1.098 at step 10,500.

The interesting part is not that it shipped. The interesting part is
*why* it shipped, given that another variant in the same sweep
(`v3_bigseq`, rank 64 at seq 8192) finished at val_loss 0.932 — a
much better number on the same parquet under the same filter. The
short answer is that "baseline" and "winner" are different roles,
and only one of them needs a deployment story.

## Corpus curation

The source is `cato:/home/cato-user/distill_corpus/plan_full.jsonl`,
8.6 GB, 180,720 rows. Each row is one teacher trajectory from
`gpt-5-nano` against a multi-bench instance — a sequence of planner
messages plus a target completion (the planner JSON the teacher
emitted). The loss is token-level cross-entropy on the *completion*
portion only. Prompt tokens get `weights=0` in the Tinker `Datum`;
they are conditioning context, not target.

`train_ultra.py` adds two filter knobs on top of the raw JSONL:
`--min-file-recall 0.5` and `--min-terminal-reward -1.0`. The v3
baseline runs with exactly those defaults. The filter keeps 178,848
rows and rejects 1,872 — a 1.0% rejection rate. The rejected rows are
trajectories that ended with no file recall *and* a strongly negative
terminal reward, which roughly means "the teacher did a bad job, the
demonstration would be miscalibrated for the student to imitate."

Notice what the filter does *not* do. It does not look at
`judge_label`. It does not gate on the per-row judge harness verdict
from `multi_bench_runs`. This matters because of an audit landed on
2026-05-11.

The 2026-05-11 pipeline-integrity audit established that
`pick_terminal_reward(RewardSource::Judge)` had been reading the
legacy NULL `result` column. Every world-model checkpoint trained off
`--reward-source judge` between the column rename and the audit was
silently regressing toward zero by construction. The judge_label
contamination class was real and was material.

What the audit *did not* affect is the SFT planner-distillation
corpus. The Tinker teacher-CE loss takes its target from the per-row
`completion` field — the JSON the teacher produced. Whether the
trajectory eventually passed the judge harness is irrelevant to the
SFT signal. The student is learning to imitate the teacher's planner
output token-by-token, not to predict the harness verdict.

So the same plan_full.jsonl that was useless for value-regression
training (because the value target was contaminated) was a perfectly
fine SFT corpus. Different loss type, different exposure. The
2026-05-11 audit found a contamination class that was selective.

That is the load-bearing point of this essay, and the rest of the
text builds toward it.

## The 3-epoch rationale

The variant name is `llama32-1b-epoch3-v3`. The `3` is a budget cap,
not a realized count. The arithmetic that turns the budget into steps
is:

- Rows kept after curation: **178,848**
- Batch size: **16**
- Bin-packing fill factor (first-fit on `max_seq_len=4096`):
  empirically ~**0.88**

The packed batch consumes roughly $178,848 / 16 = 11,178$ raw
rows worth of training per epoch, except that `train_ultra.py` packs
multiple short completions into each `max_seq_len` window, so the
*effective* number of optimizer steps per nominal epoch is

$$
\text{steps/epoch} \approx \frac{N_{\text{rows}}}{B} \cdot f_{\text{pack}}
\approx \frac{178,848}{16} \cdot 0.88 \approx 9,838
$$

A 3-epoch budget at that step rate is roughly 29,500 steps, but the
run logged ~10,892 total steps before Tinker's job slot drained.
Backing out: $10,892 / 9,838 \approx 1.11$ nominal epochs by the
crude divisor; with the more honest "raw rows / batch" denominator
ignoring packing it's $10,892 / 11,178 \approx 0.97$ epochs. The
truth is somewhere in $[0.85, 1.1]$ nominal epochs depending on
which counting convention you privilege. Call it ~0.85 with packing
overhead and the early-step warmup attributed correctly.

Why was "3 epochs" the budget if only ~0.85 actually got spent? Two
reasons.

1. Tinker has no early-stop hook. The trainer-side cosine LR schedule
   needs a target total step count to anchor the decay, and you can
   only re-anchor by killing the run. Setting the budget at 3 epochs
   lets the LR ride down smoothly through the plateau without the
   schedule "wanting" to keep stepping past convergence. If the
   budget had been set to 1 epoch and the run hit the actual val
   plateau early, the cosine would still be in its high-LR phase and
   the model would overshoot.
2. Tinker's job slot is the wall, not the budget. The run drained at
   10,892 steps because the cato-side `train_ultra.py` shard finished
   what it could in its window. The "3 epochs" was generosity that
   let the schedule behave well; the actual stopping point was
   incidental.

In practice this means the published val curve is the curve of a run
that got something between 0.85 and 1.1 effective epochs over its
filtered corpus. Not 3. The name `epoch3-v3` describes the budget
intent. The training trajectory described the realized count.

## Val_loss trajectory

The full per-checkpoint log is at
`cato:/home/cato-user/tinker_runs/llama32-1b-epoch3-v3/final.jsonl`.
The condensed story:

| step | val_loss | train_loss (band) |
|---|---|---|
| 100 | 1.479 | 1.614 |
| 1,000 | 1.276 | 1.10 |
| 3,000 | 1.150 | 0.94 |
| 5,000 | 1.114 | 0.89 |
| 7,000 | 1.103 | 0.87 |
| 9,000 | 1.100 | 0.86 |
| 10,500 | **1.098** | 0.86 |
| 10,800 | 1.099 | 0.86 |

The curve has three regimes.

**Steps 100 to ~3000.** Val drops from 1.479 to 1.150 — about 22% of
its starting headroom evaporates in the first 28% of the run. The
model is learning the planner JSON shape and the few high-frequency
tool-name patterns. Most of the win in this phase is recovering
syntactic discipline (no truncated JSON, proper tool-arg shape).

**Steps 3000 to 7000.** Val drops from 1.150 to 1.103. Slope flattens
hard. The model is learning the conditional distribution over tools
given the planner state — a much harder, lower-density signal. Train
loss settles into the 0.86–0.89 noise band.

**Steps 7000 onward.** Val sits in `[1.099, 1.108]`. Train loss
continues to nose down — into `0.85`-region transients — while val
refuses to move. This is the canonical generalization-gap
divergence: the model is starting to memorize per-trajectory
specifics that don't transfer to held-out instances. The generalization
gap at step 10,500 is `1.098 - 0.86 ≈ 0.24` natural log units, which
in token-CE terms means held-out perplexity is roughly
$e^{0.24} \approx 1.27$ times training perplexity. Modest, but real
and growing.

The "best val_loss" reported in the catalog (`1.098 @ s10500`) is the
checkpoint actually exported. The pre-plateau checkpoints in the
3000–7000 range would have been a defensible choice on
generalization-gap grounds, but two arguments lean against them:

- The plateau is genuinely flat, not just locally flat. The deltas
  inside `[7000, 10500]` are within run-to-run seed noise on val
  loss for this corpus size (we don't have multi-seed runs to put a
  hard CI on it, but the chain-deepsets sweep on the same parquet
  gave seed-to-seed val noise of ~0.005, smaller than the visible
  band here).
- Exporting at step 10,500 takes advantage of the LR schedule's full
  decay. The cosine bottoms out around step 10,500 of the budgeted
  29,500-step schedule — the optimizer is closer to a quenched fixed
  point at that ckpt than at step 7,000 where it's still moving.

Both reasons are weak; the choice could plausibly have been step
8,000 and nobody would notice. What matters is the plateau itself,
not which point on the plateau you pick.

## Comparison to the bake-off variants

The 2026-05-16 sweep ran five Tinker variants on the same plan_full
corpus. Four were launched in parallel by
`launch_variants_2026-05-16.sh`; the bare `epoch3-v3` was a sibling
sweep with `train_ultra.py` defaults. The headline numbers:

| Variant | LoRA rank | Seq | Rows kept | Steps | Best val_loss | Status |
|---|---|---|---|---|---|---|
| `tinker-3-epoch-ultra` (this run) | 64 | 4096 | 178,848 | 10,892 | **1.098** | **deployed :19320** |
| `v3_r128` | 128 | 4096 | 178,848 | 10,892 | 1.122 | deployed :19305 |
| `v3_superclean` | 64 | 4096 | 88,918 | 5,416 | 1.120 | superseded, not exported |
| `v3_bigseq` | 64 | **8192** | 178,848 | 9,386 | **0.932** | merged, *not deployed* |
| `v3_r128_short` | 128 | 4096 | 50,000 | 3,046 | 1.283 | reference, not exported |
| `v3_r256_short` | (rejected) | — | — | — | — | rank cap 128 rejected |

The reading on these numbers, variant by variant:

- `v3_r128` paid for double the LoRA rank and got back a worse number
  (1.122 vs 1.098). The hypothesis was "more capacity at a more
  conservative LR (1e-4 vs 2e-4) — better fit, no overshoot." It
  fit; it didn't overshoot. It also didn't beat the simpler
  configuration. More-rank-is-better is not the regime on this
  corpus.
- `v3_superclean` ran the strictest data filter
  (`min_file_recall=0.8, min_terminal_reward=0.5`) and kept only
  88,918 rows — about half. It hit val 1.120 at ~half the steps.
  Hyperparameter-efficient: it matched `v3_r128` on a tighter
  compute envelope. But "matched the second-place number with less
  compute" is not a winning argument against a deployment baseline
  that is already shipped at 1.098.
- `v3_r128_short` capped rows at 50,000 and hit val 1.283 — strictly
  worse on every axis. It was the data-scaling reference point.
  The deliberate purpose of this variant was to confirm
  more-data-beats-more-rank when the budget is fixed; it did.
- `v3_bigseq` is the actual winner on val_loss. Rank 64, but
  `max_seq_len=8192` instead of 4096. The hypothesis was that
  doubling the sequence-length cap would let the bin-packer fit more
  examples per window (better packing efficiency) and would let
  multi-step planner reasoning fit inside a single attention window.
  The val curve went `1.272 → 0.932` over 9,386 steps. The gap to
  the deployed baseline is **0.166 natural log units** — translating
  to roughly an 18% perplexity reduction on held-out planner JSON.
  This is not a noise win.

So why is `v3_bigseq` not the deployed baseline?

Two reasons, one technical and one structural.

**Technical.** `v3_bigseq` finished training on 2026-05-16 23:00 and
was merged into a serving-ready safetensors directory on 2026-05-17
10:13. As of 2026-05-18 it has no live `vllm serve` listener.
Deploying it would require allocating a GPU on cato, finding a free
port in `19300..19399`, launching vLLM with appropriate
`--max-num-batched-tokens` for the longer sequence cap, and
verifying the planner-recall eval on the merged weights. That work
hadn't been queued.

**Structural.** The "deployed baseline" is the reference point the
sweep variants are measured *against*. There is exactly one
production sweep baseline at any time, and changing the baseline
mid-sweep contaminates the A/B comparison. The `tinker-3-epoch-ultra`
checkpoint was shipped on 2026-05-17 18:02 because it was the
default-configuration run from the sweep. It has the
`train_ultra.py` defaults; it has the unmodified plan_full corpus
under the standard fr≥0.5/tr≥-1.0 filter; it has rank 64; it has the
default `max_seq_len=4096`. It is the reference point.

`v3_bigseq` is a *variant*. It moved one axis (`max_seq_len`) and
hit a better number. Promoting it to baseline means losing the
ability to attribute the next eval gap to either "the corpus
changed" or "the architecture changed." The conservative play is to
ship `tinker-3-epoch-ultra` as the named baseline, then run a
side-by-side planner-recall eval of `bigseq` against it, then
promote on the eval result rather than the val_loss.

This logic is unobjectionable in the abstract and gets a lot less
clean the moment val_loss disagrees with eval recall, which it often
does on this corpus.

## What "production baseline" actually buys you

The deployment story is:

1. **2026-05-17 08:45.** `tinker_train/export.py` resolved the
   `tinker://0951cfea-...:train:0/weights/shard00_final` URI to a
   signed download URL, pulled the PEFT-format tar, extracted to
   `cato:/home/cato-user/tinker_exports/llama32-1b-epoch3-v3/`. The
   step-10500 subdir was a separate export from the same run id.
2. **2026-05-17 ~10:00.** `peft.merge_and_unload()` fused the LoRA
   adapter into the base `meta-llama/Llama-3.2-1B` weights. Two
   merged-dirs were produced: a sharded 2-file safetensors at
   `tinker_merged/llama32-1b-epoch3-v3/` (presumably from the first
   export attempt) and a single-file at
   `tinker_merged/llama32-1b-epoch3-v3_step10500/`. Both are 2.4 GB
   bf16.
3. **2026-05-17 18:02.** `vllm serve` launched on the sharded merged
   dir, listening on `:19320`. The served-model-name is
   `llama32_1b_epoch3`. Flags
   `--max-num-seqs 48 --max-num-batched-tokens 24576
   --enforce-eager`.
4. `scripts/env.perseus`'s `PERSEUS_LLM_TREE_ENDPOINTS` was *not*
   pointed at `:19320`. The default planner endpoint at that
   moment was still `:19300`, the pre-sweep `llama32_1b_tinker_merged`
   from the local `3ba66240-...` Tinker run (2,000 steps, val_loss
   1.198). The new `:19320` is parked for A/B traffic from
   `tinker_eval/` — the continuous-rotation sanity loop — not for
   production routing.

So the "deployed production baseline" is, in practice, two things at
once:

- The named reference checkpoint that other sweep variants are
  measured against in the `tinker_eval` leaderboard.
- A standby planner endpoint that production traffic can be routed
  to once a planner-recall eval clears the path.

The pre-sweep `llama32_1b_tinker_merged` (the local `3ba66240` run)
is the actual default planner traffic destination. It is shipped at
val_loss 1.198, worse than the cato `0951cfea` run by 0.100, on a
smaller training-step count (2,000 vs 10,892) but the same row
filter. The gap is not large enough to force a routing switch
without evidence on a downstream metric — recall, MRR against
`fix_patch`, latency under load.

That asymmetry — "we have a better checkpoint, and we haven't moved
traffic to it" — is the actual production reality. The `:19320`
deployment is provisional. The numbers say it should win an A/B; the
A/B hasn't been run.

## Why the SFT corpus survived the 2026-05-11 contamination

The 2026-05-11 pipeline-integrity audit (T1–T9, documented in
`Claude.md`) established that the `judge_label` column on
`multi_bench_runs` had been mis-read from the legacy NULL `result`
column by `pick_terminal_reward(RewardSource::Judge)`. Every WM
checkpoint trained between the column rename and the audit had
`terminal_reward = 0` by construction. The value-regression head
trained against that target was, by construction, learning to predict
zero.

The follow-up question is: which other training pipelines were on the
contaminated path? The audit-honest answer for the Tinker SFT track
is **none**, and the reasoning is a one-liner about loss types.

The Tinker `train_ultra.py` loss is:

$$
\mathcal{L}_{\text{SFT}} = -\sum_{t \in \text{completion}}
\log p_{\theta}(y_t \mid y_{<t}, \text{prompt})
$$

The target $y_t$ is the teacher's *next token* in the completion
field of the JSONL row. The completion field was captured at trajectory
emission time, by `gpt-5-nano`, and stored verbatim. The judge label
arrives later, via the multi-swe-bench harness, and is written into a
*different* column on a *different* table.

The SFT corpus reads `messages + completion` from plan_full.jsonl.
The contamination class lived in `multi_bench_runs.judge_label`. The
two never join along the path that produces $y_t$.

The data filter does join along a downstream axis:
`--min-file-recall 0.5 --min-terminal-reward -1.0` are joins against
the multi-bench row's status fields. But the filter only *drops*
rows; it doesn't *modify* any token in a kept row's completion. A
contaminated `terminal_reward` could mean the filter is keeping or
dropping the wrong rows, but the resulting training signal on the
kept rows is intact.

Empirically: 178,848 rows passed the filter pre-audit. After the
audit the filter computes against the corrected `judge_label`
column, and `--min-terminal-reward -1.0` is generous enough that
nearly all the same rows pass — the threshold is below the
contaminated-zero value, so even the contaminated rows passed. The
filter outcome is roughly insensitive to the contamination, and the
loss target was never sensitive in the first place.

This is why "loss type" is the audit variable, not "training corpus
file." Two pipelines reading the same JSONL file can have orthogonal
exposure to a contamination class if they consume different fields
under different losses. The Tinker SFT pipeline consumed
`completion` under token-level CE. The WM value pipeline consumed
`judge_label` (via `multi_bench_runs`) under HL-Gauss regression.
The same source rows fed both. The contamination class hit one and
missed the other.

## Lesson

Loss type is a first-class audit variable. When a contamination class
is identified in a training corpus, the right next question is not
"do we throw out the corpus" but "which fields of the corpus
participate in each loss, and which losses are exposed."

The Tinker SFT pipeline survived a `judge_label` contamination
because its target token sequence (`completion`) was captured at
trajectory-emission time by the teacher and is independent of the
post-hoc harness verdict. The WM value-regression pipeline did not
survive, because its target was the harness verdict. Same JSONL.
Different exposure.

The corollary is awkward but useful: a deployment baseline that
shipped under a contaminated cohort is not, *ipso facto*, a
contaminated checkpoint. Whether the contamination reached the
weights depends on whether the contaminated field was on the loss
path. Audit per field, not per file.

And the production-baseline lesson, on a different axis: the variant
that wins on val_loss is not always the variant that gets the
deployment slot. The deployment slot has the additional job of being
the *named reference point* against which other variants are measured.
That role is taken by the configuration with the fewest moving parts
relative to the default trainer settings, not by the configuration
with the best number. The boring reference point is load-bearing
because it makes the orthogonal-axis variants measurable. The
exciting winner can wait.

`tinker-3-epoch-ultra` is the boring reference. `v3_bigseq` is the
exciting winner. Production at `:19320` runs the boring one because
the orthogonal-axis variants would not be measurable against it
otherwise. That is the entire deployment thesis in one sentence.
