> tl;dr: Four retrieval tools in the V2 catalogue accounted for 3 of
> 1,157 dev calls — 0.26% combined. The cheap thing to do is delete
> them: collapse the action space from 17 to 13, retrain a smaller
> policy head, declare victory on registry hygiene. ADR-012 refuses.
> A fixed-action-space MuZero policy learns what works from the
> training signal; pruning prejudges that learning. The four tools
> stay in `core/actions.py`, ship as regex stubs that flag their own
> approximation with a 0.85 score deflation, and graduate to real
> tree-sitter walkers when `lab/codeshape/` measures them. The
> lesson is asymmetric: adding an action is a policy-head retrain,
> removing one is too.

## 1 The four-tool family

A retrieval action space that pretends to cover "how code relates
to itself" has, at minimum, four primitives. The category isn't
arbitrary — it's the smallest set that names the four irreducible
kinds of structural code relation:

| Tool | What it answers | The graph edge |
|---|---|---|
| `symbol_lookup` | "Where is this name defined?" | `name → definition_site` |
| `references_lookup` | "Where is this symbol used?" | `definition → use_sites` |
| `callgraph_neighbors` | "Who calls this, what does it call?" | `function ↔ callers ⊕ callees` |
| `dependency_neighbors` | "What files import / depend on this?" | `module → importers` |

These map cleanly to LSP's `textDocument/definition`,
`textDocument/references`, the call hierarchy request set, and
`textDocument/typeDefinition` extended to imports. The names are
boring because the underlying graph is boring: it's the static
relation between identifiers in source code. Every code-aware tool
that has ever shipped has surfaced approximations of these four.

In V2 they live at `tools/symbol.rs` (the first three) and
`tools/file.rs::DependencyNeighborsTool`. The implementations route
service-first through retrieval-service's `/graph/*` endpoints
(tree-sitter-backed indices), falling back to lexical scans
augmented with regex-based markers — `+0.3 if snippet contains
'impl '/'fn '/'def '`, `+1.3 if snippet matches '<sym>(' or
'.<sym>('`, the multi-language `extract_dependency_terms` regex
covering seven languages. The fallback is intentionally weak; the
service path is where the real signal lives.

## 2 The usage data: 3 of 1,157

`HISTORY/24_tool_registry.md` records the cold truth. Across all
local query-progress traces between 2026-04-23 and 2026-05-16,
filtered to queries that fired any tool observation event:

| Tool | Calls | % of 1,157 |
|---|---:|---:|
| `hybrid_search` | 462 | 39.4% |
| `search_text` | 230 | 19.6% |
| `search_path` | 219 | 18.7% |
| `diff_pattern_scan` | 111 | 9.5% |
| `broad_scan` | 61 | 5.2% |
| `open_file` | 55 | 4.7% |
| `snippet_extract` | 9 | 0.8% |
| `repo_stats` | 7 | 0.6% |
| `symbol_lookup` | **3** | **0.3%** |
| `references_lookup` | **0** | **0.0%** |
| `callgraph_neighbors` | **0** | **0.0%** |
| `dependency_neighbors` | **0** | **0.0%** |
| (8 other tools at 0%) | — | — |

The four symbol-graph tools combined: 3 calls out of 1,157. That's
0.26% — three orders of magnitude below `hybrid_search`. By any
straightforward usage metric, they are dead weight. The graph
endpoints went live with the cleanup-250-loc audit chain on
2026-04-24; in the three weeks of dev traffic since, the planner
essentially never chose them.

The local-trace caveat from HISTORY/24 is honest: this is a dev
workload, biased toward queries against the Perseus repo itself —
the kind of question the planner answers by finding a snippet, not
by walking a call graph. Production sweep numbers via
`scripts/action_dist.sh` on the engram Postgres `tool_events` table
could tell a different story. But the dev data is what we have, and
the dev data is unambiguous: the planner is not picking these
tools.

## 3 The pruning case (and why it's wrong)

The cheap, obvious move: delete the four tools. Concretely that
means:

1. Remove the `Arc::new(SymbolLookupTool)` lines from
   `default_registry()` in `tools.rs`.
2. Drop the corresponding `ToolName` variants from `schema.rs`.
3. Regenerate the planner prompt without their catalogue entries.
4. Retrain the policy head over 13 actions instead of 17.

The case for pruning is straightforward. Smaller action space →
shorter prompt → cheaper LLM call → faster planner convergence.
Smaller policy head → faster supervised training. Fewer dispatch
paths to maintain. A registry of 13 tools the planner actually uses
is cleaner than 17 with four that are essentially extinct.

ADR-012 is explicit about why this is wrong. The decision document
captures perseus's identity as a MuZero application for code
retrieval, and lays out the action space — all 17 tools — in the
opening context block. Then it lists six things "V2 got wrong (and
is not an indictment of the architecture)." The third bullet:

> **17 tools coexisted with 0.26% combined usage of 4 of them**.
> The fix is to keep them in the action space (the policy will
> learn what works) — not amputate them.

This is the load-bearing argument. The policy head is the
mechanism by which dead tools become dead tools. If, in training,
the policy converges on a distribution that never emits
`callgraph_neighbors`, then the deployed policy never emits
`callgraph_neighbors` — and the operational behavior is identical
to having deleted the tool. The difference is reversibility. Under
pruning, "let's revisit callgraph for the refactoring workload"
requires re-introducing the action, retraining the policy head from
scratch, and re-validating against the cohort that the policy
originally trained on. Under keep-anyway, it requires nothing — the
policy head already has a (small) prior over `callgraph_neighbors`;
training data from a new workload tilts that prior; production
behavior changes without code change.

Pruning prejudges what the policy can learn from data it has not
yet seen. Keep-anyway is the option-value preservation.

## 4 The dev-vs-production workload mismatch

The third reason pruning is wrong is that the data establishing
"these four tools are dead" is the wrong data. Local-trace queries
look like:

- "How is the LLM planner timeout configured?"
- "Where does `pick_terminal_reward` decide value targets?"
- "Why does `confirm_stop` reject branches?"

These are snippet-shaped queries. The right tool is `hybrid_search`
(which returns the relevant chunk with line range and language tag)
followed by `open_file` or `snippet_extract` to read the surrounding
context. The planner converges in 2-3 calls. `callgraph_neighbors`
adds nothing because the planner already has the snippet — it
doesn't need to walk the call graph.

Production debugging workflows are not snippet-shaped. Consider the
kinds of queries that arise when an engineer is doing real work:

- "If I change this function signature, what breaks?" — that's
  `references_lookup` on the function name, recursively.
- "I'm refactoring the auth module. What other modules touch it?"
  — that's `dependency_neighbors` on the module path.
- "I'm tracing a regression in `compress_block`. What's the call
  chain from the public API down to here?" — that's
  `callgraph_neighbors` walked both ways from the leaf back to the
  entry point.

For these workloads, the symbol-graph tools are the right tool —
hybrid_search isn't even close. Hybrid_search optimizes for
"surface a relevant snippet given a free-form question."
Callgraph_neighbors optimizes for "give me the structural
neighborhood of an identifier I already have." Different problems,
different answers.

The mistake in deciding to prune based on V2 dev traces is treating
the dev distribution as a stand-in for the production distribution.
The fix is not "extrapolate harder from the local traces." The fix
is "train the policy on data from the workload the planner will
actually see." Pruning fixes the symptom and removes the mechanism
that would let the policy correct itself when the real workload
arrives.

## 5 The regex-stub state of play

The other reason pruning would be premature: the four tools are
not really being tested. The Python rewrite in
`perseus/core/retrieval/tools/codeshape.py` ships them as regex
stubs, explicit about their own approximation. The file's
docstring is unusually candid:

```python
"""Tree-sitter-backed actions: SymbolLookup, ReferencesLookup,
CallgraphNeighbors, DependencyNeighbors.

v0.1 ships with regex-only fallback impls so the action space is
complete (the policy head emits prior over 17 actions). When the
proper tree-sitter walker lands (graduates from lab/codeshape/),
these classes get swapped in place — the registry shape doesn't
change.
"""
```

The implementations themselves are deliberately thin.
`CallgraphNeighbors.run` is literally:

```python
class CallgraphNeighbors:
    name = ToolName.CALLGRAPH_NEIGHBORS

    async def run(self, call, *, repo_root, index_id):
        # v0.1: same as references_lookup. Promoted to true
        # callgraph when lab/codeshape/ tree-sitter walker
        # graduates per ADR-006.
        symbol = str(call.args.get("symbol", "")).strip()
        limit = int(call.args.get("limit", 20))
        if not symbol:
            raise PerseusValidationError("callgraph_neighbors requires 'symbol'")
        hits = await _grep(symbol, index_id, limit)
        # Slight score-deflation to mark this is "neighborhood"
        # not "true callgraph".
        for h in hits:
            h.score *= 0.85
        return ToolEvidence(hits=hits, outcome_score=_outcome(hits))
```

Two things to notice. First, `CallgraphNeighbors` is currently
identical to `ReferencesLookup` except for the 0.85 score
multiplier. There is no callgraph walk; there's a grep over symbol
occurrences. Second, the 0.85 is not a tuned hyperparameter — it's
a deliberate signal in the score that says "I'm an approximation,
deflate me when ranked against tools whose evidence is more
structural." A trained policy that learned `callgraph_neighbors`
was useless against the regex stub would be learning that regex-
based callgraph is useless, not that callgraph-the-concept is
useless. Pruning based on V2 data conflates the two.

The graduation path is documented: when `lab/codeshape/` produces
a tree-sitter walker that actually returns callers and callees
distinct from a textual grep, the four classes get swapped in
place. Per ADR-006's graduation rule, the swap requires a
measurement: extend the operator benchmark with refactor /
breaking-change-impact cases, run baseline (current regex stub),
run alternative (tree-sitter walker), promote only if alternative
wins. The registry shape doesn't change. The policy head doesn't
retrain unless the action distribution changes meaningfully.

## 6 Asymmetric costs of action-space surgery

The case for keep-anyway gets stronger when you look at the
asymmetric cost of modifying the action space at all.

**Adding action 18.** Suppose two months from now we want a new
tool — say, `git_blame_window`, which surfaces the most recent N
commits that touched a line range. Adding it costs:

1. New `ToolName` variant and `specificity()` weight (15 LOC).
2. New `Tool` impl (~150 LOC).
3. Register in `default_registry()` and bump the test count.
4. **Retrain the policy head** — the head emits a categorical
   distribution over actions, so the output dimension changes
   from 17 to 18. Every checkpoint trained on 17 is incompatible.
   This is the load-bearing cost. Depending on cohort size, it's
   a multi-day training run plus eval.
5. Update the planner prompt catalogue.
6. Update the HISTORY docs and `agents.md` Tooling Surface bullet.

The 150 LOC of implementation is the cheap part. The policy-head
retrain is the expensive part. Adding an action is architectural
surgery, not a registry edit.

**Removing action 18.** Suppose we shipped it, watched it sit at
0% for a month, and decided to prune. Removing it costs:

1. Delete the `Arc::new` line (1 LOC).
2. Delete the `ToolName` variant and the impl (~150 LOC).
3. **Retrain the policy head** — output dimension changes from 18
   back to 17. Every checkpoint trained on 18 is incompatible.
4. Update the prompt catalogue.
5. Update the HISTORY docs.

Removing an action is the same architectural surgery as adding
one. The cost isn't in the code; it's in the policy head
retrain. There is no cheap-remove path that preserves the
checkpoint.

This is the asymmetry that pruning advocates miss. "Removing dead
tools is hygiene" treats the action space like a Python module
registry — a thing you can edit freely because the only cost is
deleted lines. The action space of a fixed-action MuZero policy
is a typed enum baked into the policy-head output layer; editing
it is a model surgery.

Given that the cost of removing a tool is the same as the cost of
adding one, the question reduces to: under what circumstances is
it worth paying the surgery cost to remove a tool whose only crime
is being unused? The answer is "never, unless the tool actively
hurts the planner" — and a tool the planner never picks doesn't
hurt anything. It sits in the catalogue with zero visit mass.

## 7 The lesson, restated

The frustrating thing about a fixed-action-space MuZero is that it
constrains research velocity. Adding action 18 is a policy
retrain. You feel this every time you want to try a new tool
shape. The temptation is to fix that constraint by curating the
action space aggressively — keep it small, swap actions in and out
freely, optimize for "minimum viable catalogue."

That trade is wrong. The constraint is real, but the optimization
that responds to it is not "prune unused actions." It's "design
the action space well once, then commit." Symbol-graph belongs in
the action space because the category is one of the four
irreducible structural-code-relation primitives. Whether the
planner picks the tool on dev workloads is a policy-head training
signal question, not an action-space-design question.

Pruning is the right move when an action is actively misleading —
returning hits that confuse the planner, dominating the prior in
ways the policy can't recover from. The symbol-graph tools are
not misleading. They're quiet. They sit at 0.26% combined usage,
return regex-quality results when the planner does call them, and
flag their own approximation with the 0.85 deflation marker. The
policy head, given enough production data from a workload where
callgraph actually matters, will tilt toward them.

The work that matters isn't removing actions from the catalogue.
It is:

1. Building the tree-sitter walker in `lab/codeshape/` so that
   when the planner does call these tools, the evidence is
   structural (real callers / callees) instead of regex hits.
2. Generating training data from production-shaped workloads
   (refactor planning, breaking-change impact, regression
   tracing) so the policy head sees examples where symbol-graph
   tools win.
3. Extending the operator benchmark with cases that require the
   structural relation — so the graduation measurement under
   ADR-006 has somewhere to land.

That work belongs in the policy head's training signal and in the
tool implementations themselves. The action-space catalogue is
the wrong place to optimize. ADR-012's keep-anyway argument is
the right policy: 17 actions, all of them, the policy learns what
works.

## 8 Where this lives now

The four tools sit at:

- `perseus/core/retrieval/tools/codeshape.py` — v0.1 regex stubs,
  awaiting the tree-sitter graduation from `lab/codeshape/`.
- The 17-entry `ToolName` enum in `perseus/core/actions.py` —
  unchanged from V2's catalogue per ADR-012.
- The catalogue block in the planner system prompt — all four
  listed at specificity 1.20, "high" tier, with one-sentence
  semantic disambiguations.

The 0.26% usage number is recorded in `HISTORY/24_tool_registry.md`
and `HISTORY/39_tool_implementations.md` as the V2 cohort's
baseline. The next cohort that ships will be measured against
that baseline; if production traffic continues to put callgraph
at 0%, the tools stay anyway, because the cost of keeping them is
zero and the option value of having them when refactor workloads
arrive is non-zero.

The action space is not the place to chase efficiency. The
policy head is.
